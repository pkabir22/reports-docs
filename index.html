Built & automated by Mahfuz Kabir
This is ABL's most critical and complex daily report — a real-time command center for the Treasury team's liquidity management, covenant compliance, and capital markets decision-making. It was significantly re-engineered by Mahfuz to reduce manual effort, eliminate error-prone steps, and cut production time through Power Query automation and VBA macro development.
Purpose & Overview
A comprehensive daily snapshot of ABL's capital structure and liquidity position. Covers real-time warehouse (WH) utilization, liquidity runway, funding mix, and covenant compliance across all warehouses and securitization shelves.
Report Sections
Section 1 — Capital Structure & Liquidity Snapshot
Daily overview of total indebtedness, available capacity, leverage ratios, blended cost of capital, and ending cash position.
Section 2 — Securitizations (2 RTL Shelves)
Balances, cash held in payment/accumulation accounts, UPB surplus/shortfall, and associated funding costs per securitization shelf.
Section 3 — Warehouse Financing (Multi-Bank)
Facility limits, collateral pledged, outstanding balances, PVE, funding costs, and current cash-covenant status per lender/warehouse.
Section 4 — Daily Cash Flow Tracker & 29-Day Projection
Ties operating cash & PVE to funding activity, capital markets actions, payoffs, payroll, ending liquidity, and covenant coverage. Projects ending liquidity for the next 29 work days.

Step-by-Step Production Process
  1. Refresh Dynamics CRM Power Queries — open the master workbook and run "Refresh All" to pull the latest data into the Bridge, Term, 2024-RTL1, 2025-RTL1, and 1Sharpe tabs.
  2. Load CFSB Daily Summary — confirm Treasury has uploaded the latest Daily RPC Sheet to Dropbox. Load via Get Data › From Excel Workbook.
  3. Load Nomura EOD Report — confirm upload to Dropbox, load the Results tab.
  4. Enter Mizuho EOD figures — open the Mizuho end-of-day email and manually enter figures into the 1_Pager.
  5. Calculate and enter Payoffs — open each Investor Notification Deposit PDF. Locate Principal Payment column (6th from right), take 22.5% of that figure, sum all values → cell L23.
  6. Enter Teams channel data — check Teams for DSCR Sale/Repo figures, RTL/Trust activity, and other capital/funding activity.
  7. Enter Operating Cash — log into CFSB portal: SUM(Acct 5247 + Acct 5239 + Mor Dutch Esc + Acct 5352).
  8. Enter RTL1 PA & AA figures — log into USB Pivot portal and enter Payment Account and Accumulation Account balances for 2024-RTL1 and 2025-RTL1.
  9. Verify formula propagation — confirm all entries have flowed through Excel formulas into the 1_Pager correctly.
  10. Validate covenant & liquidity figures — review covenant cells for flags, cross-check ending cash, WH utilization, and PVE.

Sub RefreshAllQueries()

▼
This dashboard is powered by a single large SQL mart table — marts.kpi.kpi__dscr_underwriting — built by Mahfuz from scratch. The table was constructed first as a multi-CTE CREATE OR REPLACE TABLE statement in Databricks SQL, pulling from the CRM staging layer. Once the table was live, Power BI was connected to it with a simple SELECT * FROM marts.kpi.kpi__dscr_underwriting to drive the dashboard.
How It Was Built — Two-Step Approach
Step 1 — Build the SQL Mart Table
Mahfuz wrote a single large SQL query (196 lines, 6 CTEs) as a CREATE OR REPLACE TABLE statement. This query pulls the DSCR loan pipeline from staging.dynamics.term and enriches every loan with milestone dates and stage-transition context by mining staging.dynamics.audit. The full query is documented in the SQL section below.
Step 2 — Connect Power BI
Once the mart table was live and validated, Power BI was pointed at it with a single query:

SELECT * FROM marts.kpi.kpi__dscr_underwriting

All dashboard visuals, filters, and KPIs are built directly on top of this table.
Purpose & Overview
Tracks DSCR loans from pipeline entry through underwriting milestones to close. Surfaces how long loans spend in each underwriting stage, how many times loans are re-submitted for Final UW, and what stage loans move to after Initial and Final Underwriting — enabling the team to identify bottlenecks and measure UW throughput.

create or replace table marts.kpi.kpi__dscr_underwriting as (

with cte as (
    select
        new_termloanid          as guid,
        new_statusname          as status,
        statuscodename,
        new_activedealstage name as deal_stage,
        new_subject             as subject,
        new_uwlookupname        as uw,
        owneridname             as lo,
        new_processorname       as processor,
        new_targetloanbuyername as target_loan_buyer,
        --cast(createdon as date) as file_creation_date,
        cast(new_trueclosingdate as date) as closing_date,
        new_totalloanamount     as loan_amount
    from staging.dynamics.term
    where
        (
            new_statusname = 'Won'
            and new_trueclosingdate >= '2025-01-01'
        )
        or
        (
            statuscodename != 'Inactive'
            and new_statusname != 'Lost'
            and new_activedealstage name in (
                'Processing',
                'Initial Underwriting',
                'Conditionally Approved',
                'Final Underwriting',
                'Clear To Close'
            )
        )
    order by closing_date desc
),

before_final_uw as (
    select * from (
        select
            cast(createdon as date) as date_stage_before_final_uw,
            objectid,
            old_name as old_stage,
            case
                when new_name is null then 'Post Initial UW Stage'
                else new_name
            end as new_stage,
            row_number() over(
                partition by objectid, new_value
                order by createdon
            ) as n
        from staging.dynamics.audit
        inner join cte on cte.guid = audit.objectid
        where
            objecttypecode = 'new_termloan'
            and old_name = 'Final Underwriting'
            and new_name in (
                'Conditionally Approved',
                'Processing',
                'Initial Underwrite',
                'Prelim UW',
                'Ready for Credit Pull'
            )
    ) where n = 1
),

Final SELECT — Joined Output 5 LEFT JOINs · filtered + ordered
last_update via from_utc_timestampfilter: 1st_lts_sent_date ≥ 2024-06-01ORDER BY 1st_final_uw DESC
▼ View code 
select
    from_utc_timestamp(
        least(
            (select max(createdon) from staging.dynamics.audit),
            (select max(pulled_on)  from staging.dynamics.term)
        )
    , 'America/New_York') as last_update,
    cte.*,
    fl.1st_lts_sent_date            as 1st_lts_sent,
    fi.1st_initial_uw_mark_date     as 1st_initial_uw,
    bi.date_stage_before_initial_uw,
    bi.new_stage                    as stage_before_initial_uw,
    ff.1st_final_uw_mark_date       as 1st_final_uw,
    bf.date_stage_before_final_uw,
    bf.new_stage                    as stage_before_final_uw

from cte
left join first_lts_sent    as fl on fl.objectid = cte.guid
left join first_initial_uw  as fi on fi.objectid = cte.guid
left join first_final_uw    as ff on ff.objectid = cte.guid
left join before_initial_uw as bi on bi.objectid = cte.guid
left join before_final_uw   as bf on bf.objectid = cte.guid

where fl.1st_lts_sent_date >= '2024-06-01'
    --or cte.closing_date >= '2025-01-01'
order by 1st_final_uw desc

)

Key terms used across ABL's treasury and finance reports.

TermDefinition
WH / WarehouseShort-term credit facility from a bank used to fund loan originations before they are sold or securitized.
PVEPurchase Value Excess — the amount by which the market/purchase value of collateral exceeds the outstanding warehouse balance.
UPBUnpaid Principal Balance — the remaining principal owed on a loan or pool of loans.
RTL ShelfResidential Transition Loan securitization shelf — a registered structure used to issue ABS backed by short-term bridge/fix-and-flip loans.
SecuritizationProcess of pooling loans and issuing securities backed by those loans to capital markets investors.
CovenantContractual requirement in a credit facility. Breach triggers lender remedies.
DSCRDebt Service Coverage Ratio — a loan type for investment properties assessed on rental income rather than borrower income.
Initial UWFirst underwriting review stage a DSCR loan enters. First date marked captured as 1st_initial_uw.
Final UWFinal underwriting stage before a DSCR loan clears to close. First date marked captured as 1st_final_uw.
LTS / Term Sheet SentStage when the Loan Term Sheet is formally sent to the borrower. Captured as 1st_lts_sent.
Audit Logstaging.dynamics.audit — records every field-level change to a loan record, including stage transitions.
CTECommon Table Expression — a named subquery used to modularise complex SQL logic before a final SELECT.
Mart tableA curated, production-ready table in the marts schema, consumed by Power BI dashboards.
PA / AAPayment Account / Accumulation Account — trust accounts within a securitization structure.
1_PagerThe master summary tab/output of the Daily Liquidity Report.
